Design panel

A fixed-position dev panel for checking layout, alignment, and rhythm. Shadow-DOM web component, keyboard-driven, zero dependencies.

Overview

The design panel is a <design-panel> web component that slides in from the right edge of your browser window. It provides visual overlays and toggles for verifying alignment to the baseline grid, checking spacing, and debugging layouts. Chrome lives in Shadow DOM so it stays isolated from your page styles.

Opening the panel

Press T anywhere on the page to toggle the panel. Or click the floating action button at the bottom-right of the viewport. Keyboard shortcuts are input-aware: typing in a form field won't trigger them.

Keyboard shortcuts work whether the panel is open or closed. Press any tool's shortcut key to toggle it from anywhere on the page.

Tabs

The panel has four tabs: Guides, Typography, Colours, and Theme. All four ship active: Guides toggles visual overlays, Typography edits the type scale, Colours edits ramps and schemes, and Theme saves the current runtime state as a named bundle you can commit to the repo.

Tools

Each tool can be toggled by clicking its button in the Guides tab or by pressing its keyboard shortcut.

Key Tool Description
B Baseline Display horizontal lines at every baseline increment. Use this to verify text and spacing align to the vertical rhythm. Subdivision lines (lighter) show fractional increments.
C Columns Show a column grid overlay across the viewport. Useful for checking layout alignment to a column-based grid. Configure column count, gap, and margins in Settings.
D Dark mode Toggle between light and dark mode, overriding system preference.
G Grids Outline all .grid layout primitives on the page and display the detected column count.
M Margins Highlight the horizontal page margins (the space on either side of your content).
O Outlines Add outlines to all elements so you can see layout boundaries and nesting.
X Hide backgrounds Strip all background colours and images. Useful for seeing through coloured sections to check underlying layout.
R Redact Apply a "redacted" font to prose content, replacing readable text with abstract shapes. Useful for evaluating layout and hierarchy without being distracted by content.

Settings

The Guides tab also exposes settings for the column and baseline overlays.

Columns

Set the number of columns for the column grid overlay (1-24). This is purely a visual reference; it doesn't affect your actual CSS grid layouts.

Subdivisions

Choose how many subdivisions to show between baseline lines:

  • 1: No subdivisions, only main baseline lines
  • 2: Half-line subdivisions (default)
  • 3: Third-line subdivisions
  • 4: Quarter-line subdivisions

Gap

Set the gap between columns in the overlay. Labels match the panel select exactly:

  • Default: var(--gutter) (default)
  • 0: No gap
  • 1px: var(--line-1px)
  • 0.25: var(--line-025)
  • 0.5: var(--line-05)
  • 1: var(--line-1)
  • 2: var(--line-2)

Margin mode

Choose which margin mode the column overlay uses:

  • Section: Matches the responsive margins of .section (default)
  • Full-bleed: No margins, edge to edge
  • Wide: Matches the narrower margins of .section-wide

Colours

The Colours tab contains two editors: the ramp matrix and the schemes editor.

Ramp matrix

Each of the six families (Blue, Red, Orange, Yellow, Green, Grey) expands into a settings row with three controls:

  • Anchor colour — a <input type="color"> seeded from the family's current --color-{family}-500 token. Pick a new anchor and all 11 steps regenerate live via the OKLCH ramp algorithm.
  • Anchor step — which step the anchor should pin to (50, 100, 200, ..., 950). Changing this shifts the lightness curve so the anchor sits at the chosen step's L target.
  • Chroma intensity — a 0-100 slider that scales the default chroma profile. At 100 the algorithm produces the canonical saturated ramp; at 0 every step collapses to a neutral grey.

Regeneration is entirely client-side. No network requests fire when you drag a slider or pick a new colour; the math runs in src/js/color/ramp.js, which is a line-for-line port of Assembly's backend/internal/color/ramp.go.

Schemes editor

Three scheme preview cards (Default, Subtle, Accent) each expose independent Light and Dark overrides for the five semantic tokens: --color-bg, --color-fg, --color-accent, --color-muted, --color-subtle. Each selection is wired through a dual-emit router so that Default-scheme edits cascade at :root/.dark-mode while Subtle/Accent edits scope to their .scheme-X classes.

Ramp and scheme persistence

Ramp settings live in localStorage['design-panel:ramps'] as per-family {anchorHex, anchorStep, chroma} entries. Scheme selections live in localStorage['design-panel:schemes']. Both restore automatically on reload; ramps re-run the generator so the regenerated tokens match what you saw before the reload.

Typography

The Typography tab is the simplest editor — nine form inputs that each write one CSS custom property on :root. No backend, no network, no algorithm. Sections from top to bottom:

Scale

The scale-ratio slider writes --type-scale-ratio (range 1.1 to 1.618, step 0.001). Seven preset buttons jump to the standard musical-interval ratios:

  • 1.125 — Minor second
  • 1.2 — Minor third
  • 1.25 — Major third
  • 1.333 — Perfect fourth
  • 1.414 — Augmented fourth
  • 1.5 — Perfect fifth
  • 1.618 — Golden ratio

Clicking a preset writes the exact value to the slider; the matching button gets aria-pressed="true" until you nudge the slider off the value.

Base size

Min and Max number inputs write --text-base-min and --text-base-max (px, unitless in the token). The clamp() wrappers in typography-scale-auto.css turn these into the fluid base font size.

Line height

Body and Heading range sliders (1.0 to 2.0, step 0.05) write --line-height-ratio-body and --line-height-ratio-heading. These cascade into all per-size --line-height-* tokens that use them.

Fonts

Body and Heading text inputs write --font-body and --font-heading. Inputs intentionally start blank so the cascade default (var(--font-sans)) keeps applying; the placeholder text shows what's currently inherited. Type a font stack like Georgia, serif to override; clear the input to revert.

Baseline

Min and Max number inputs write --line-height-min and --line-height-max. The computed --line baseline reads these, so the Guides tool's baseline overlay re-aligns the moment you adjust them.

Persistence

All nine input values persist to localStorage['design-panel:typography'] with a 200ms debounce. On reload the controller restores stored values; missing keys hydrate from the cascade (getComputedStyle() against :root). The Theme tab (see below) promotes these runtime edits into named, version-controllable JSON bundles stored in public/themes/.

Accessibility

Range inputs update aria-valuetext on every change so screen readers announce the current value with context (e.g. "Scale ratio 1.25, Major third"; "Body line height 1.5"). Min/Max number-input pairs are grouped in <fieldset> with a <legend> so the group name is announced when each input gains focus.

Theme tab

The Theme tab is the canonical persistence layer for design-panel state. Where Typography and Colours edit runtime values, the Theme tab saves those values as named JSON bundles in public/themes/ that you can commit to your repo, share between machines, and switch between during development.

What a theme is

A theme is a JSON file at public/themes/<id>.json. Each bundle carries the current typography and scheme state plus metadata:

{
  "schemaVersion": 1,
  "id": "thm_dusk",
  "name": "Dusk",
  "description": "Serif headlines, tighter scale",
  "isDefault": false,
  "createdAt": "2026-04-21T00:00:00Z",
  "updatedAt": "2026-04-21T00:00:00Z",
  "typography": {
    "--type-scale-ratio": "1.2",
    "--font-body": "Georgia, serif"
  },
  "schemes": {
    "default": { "light": {}, "dark": {} },
    "subtle":  { "light": {}, "dark": {} },
    "accent":  { "light": {}, "dark": {} }
  }
}

Empty typography and schemes objects mean "use cascade defaults" — activating such a theme calls removeProperty on every override.

UI

The tab has three sections:

  • Current theme — shows the active theme name, with Save and Reset buttons. Save writes the current runtime state to the active theme's JSON file; Reset re-applies the active theme's last saved state, discarding any unsaved edits.
  • Saved themes — lists every theme in public/themes/. Default is pinned first with a read-only badge. Non-active rows have an Activate button; non-default rows also have a Delete button (wrapped in a confirmation dialog).
  • New theme — a labeled form that creates a new bundle by snapshotting the current runtime state. Names are validated 1-80 characters and must be unique.

Default theme protection

The default theme (thm_default) is read-only. When it is active, the Save button is disabled, Typography and Colours inputs all carry disabled, and the default row has no Delete button. The Vite endpoint also refuses DELETE /__dp/themes/thm_default with HTTP 403. To edit starting from defaults, create a new theme first.

Dev-only persistence via Vite

The Theme tab talks to a small dev-only middleware registered in vite.config.js as designPanelThemesPlugin. It exposes three endpoints on the Vite dev server:

  • GET /__dp/themes — list all bundles
  • PUT /__dp/themes/:id — upsert a bundle (validates id regex, name length, property-key regex, body size; writes atomically via temp + rename)
  • DELETE /__dp/themes/:id — delete; refuses isDefault: true

The plugin uses apply: 'serve' so production builds never include it. Production users see the Theme tab as a static shell — they can read whatever JSON bundles are on disk but cannot save or delete. Localhost-only guard; X-Forwarded-For is ignored.

Cross-controller activation

Switching themes coordinates three controllers. The sequence is:

  1. The theme controller flushes pending debounced saves in Typography and Colours via window.__dpTypographySave.flush() and window.__dpColorsSave.flush().
  2. It calls window.__dpSchemeReset() to tear down any runtime scheme overrides.
  3. It overwrites localStorage['design-panel:typography'] and ['design-panel:schemes'] with the new theme's values (translating from bundle shape to storage shape at the seam).
  4. It calls removeProperty on any :root typography custom property not carried by the new theme.
  5. It writes localStorage['design-panel:active-theme-id'].
  6. It dispatches design-panel:reactivate on document. Typography and Colours controllers listen for this event and re-run their applyState() functions.
  7. It updates the Theme tab UI (active row, current name, status announcement).

Accessibility

The name input is labeled (<label for>) and carries maxlength="80" + required. Validation errors set aria-invalid="true" on the input and route descriptive text through aria-describedby. The save/activate/delete status messages announce via a visually hidden <output role="status" aria-live="polite"> region. The active theme row carries aria-current="true" so screen readers announce the current selection. Delete is wrapped in confirm() to prevent accidental loss.

Persistence

All panel settings and toggle states are saved to localStorage. When you reload the page or navigate to a different page, your tools remain active and settings are preserved.

To reset to defaults, clear your browser's local storage for this site or use browser dev tools to delete the design-panel:state key.

Routing tool classes to a different target

By default the panel writes tool classes (dark-mode, show-baseline, etc.) to the <body> element. Pass an overlay-target attribute on the custom element to route them somewhere else:

<design-panel overlay-target=".workspace"></design-panel>

Handy if you're prototyping an app shell and want the overlays to stop at a layout boundary.

Tips

Checking baseline alignment

Enable B Baseline and scroll through your page. Text baselines should sit on the grid lines. If text appears between lines, check your line-height values or spacing utilities.

Debugging grid layouts

Use G Grids to see exactly how many columns each .grid element has at the current viewport size. This is especially useful for auto-fit grids that change column count responsively.

Evaluating visual hierarchy

Enable R Redact to blur text content and focus on shapes, sizes, and spacing. This helps you see if your visual hierarchy works without being influenced by the actual words.

Checking content structure without colour

Toggle X Hide backgrounds to strip all background colours. This reveals bare content hierarchy and exposes where sections begin and end without relying on colour.

Verifying responsive margins

Enable M Margins and resize your browser. The margin overlay responds to the same breakpoints as .section, so you can verify your content aligns with the margin system.
