Design process

A practical workflow for building prototypes with Live Wires.

Philosophy

Live Wires is built for fast iteration. The workflow prioritizes getting ideas into the browser quickly, then refining through progressive enhancement.

The key insight: semantic HTML looks good by default. Start with structure, not styling. Add classes only when you need specific control.

The four-phase workflow

Phase 1: Rapid prototyping

Build structure first. Write semantic HTML without worrying about visual details.

  1. Create your page in /public/
  2. Write semantic HTML—headings, paragraphs, lists, forms
  3. Add layout primitives—.stack, .grid, .sidebar
  4. Use HTML includes for repeated elements (header, footer, nav)

At this stage, your page should be functional and readable. The baseline styles handle typography, spacing, and color automatically.

Phase 2: Configure design tokens

Once structure is solid, customize the system-wide settings.

typography-base.css
Font stacks, base size, line height ratio. Changes here ripple through the entire system.
color.css
Color palette, semantic colors, schemes. Define your brand colors here.
spacing.css
The sacred --line variable. All spacing derives from this.
borders.css
Border radius scale for consistent roundness.

Token changes are global. Set them once, use them everywhere.

Phase 3: Art direction with utilities

Add utility classes for specific visual adjustments.

<!-- Before: semantic HTML with layout -->
<article class="stack">
  <h2>Article Title</h2>
  <p>Content...</p>
</article>

<!-- After: adding art direction -->
<article class="stack scheme-warm px-3 py-4">
  <h2 class="text-3xl">Article Title</h2>
  <p class="text-lg measure">Content...</p>
</article>

Use utilities sparingly. If you're adding the same utilities to every element, consider adjusting the token or creating a component.

Phase 4: Polish and review

Use the style guide as a checklist to verify consistency.

  • Typography—Check the Elements page for heading hierarchy
  • Spacing—Enable the baseline grid overlay (.show-baseline) to verify vertical rhythm
  • Color—Review color contrast and scheme consistency
  • Components—Cross-reference with the Components page
  • Responsive—Test at various viewport sizes

Development tools

Live Wires includes development utilities to help verify your work.

Baseline grid overlay

<body class="show-baseline">

Shows the baseline grid to verify vertical rhythm alignment. All spacing should snap to lines.

Column overlays

<body class="show-columns-2">
<body class="show-columns-3">
<body class="show-columns-4">

Visualize column layouts during development.

Dark mode toggle

<body class="dark-mode">

Force dark mode regardless of system preference.

These can also be toggled via keyboard shortcuts defined in dev-tools.js.

Using the style guide

The /guide/ folder serves multiple purposes:

Documentation
Reference for available components, utilities, and patterns.
Component library
Copy HTML snippets directly from the examples.
Testing ground
A pre-built site to verify token changes affect things correctly.
Starting point
Duplicate and modify guide pages as templates for your prototype.

When you change design tokens, check the guide pages to see the effects across all components and elements.

Tips for fast iteration

Start with content

Don't design empty boxes. Use real (or realistic) content from the start. This reveals layout problems early.

Embrace good defaults

Before adding a class, ask: "Does this actually need styling?" Often the answer is no. Semantic HTML with default styles is usually enough.

Use the cascade

Apply utilities to parent elements when possible. A .scheme-warm on a section affects all children.

Inspect while you work

Keep browser DevTools open. Inspect computed values. Check the cascade layers panel to understand specificity.

Commit early, commit often

Prototype quickly means you'll try things that don't work. Use Git to checkpoint working states.

When to create components

Live Wires encourages using utilities and layout primitives over custom components. But sometimes a component is the right choice.

Create a component when:

  • You're repeating the same utility combination 3+ times
  • The pattern has internal structure (children that relate to each other)
  • You need container queries for responsive behavior
  • The pattern will be used across multiple pages

Keep using utilities when:

  • Each instance is slightly different
  • The styling is a one-off adjustment
  • You're still exploring options

See the Code style guide for component naming conventions.
