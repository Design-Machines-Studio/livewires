Live Wires 2026 Project Specification
For Claude Code Implementation

Project Context
Project Name: Live Wires 2026
Type: CSS Framework + Prototyping System + CMS Integration
Purpose: Modern evolution of a prototyping-first web design workflow
Primary Use Case: Editorial/magazine-style websites with art direction flexibility
Philosophy (Critical to Understand)
Live Wires is not a component library like Bootstrap or Tailwind. It's a prototyping-to-production workflow that:

Starts with HTML prototypes (semantic, minimal classes)
Systematizes patterns into a style guide
Layers art direction via utility classes
Integrates with CMS (primarily Craft CMS)

Core principle: Good defaults + additive art direction. Pages should look decent with zero classes, great with thoughtful class application.

/templates/_components/utilities/ - Class generation macros
Study but don't copy: the baseline system, grid utilities, and component patterns


Technical Specifications
Technology Stack
Required:

PostCSS (for CSS processing)
CSS Nesting (native, no preprocessor)
CSS Cascade Layers (native)
Modern CSS features (container queries, custom properties, :has(), etc.)

Not Required:

SCSS/Sass (migrate away from preprocessing)
JavaScript frameworks (vanilla JS only if needed)
Build complexity (keep it simple)

Development Tools:

Vite (for dev server + building)
Modern normalize/reset
No jQuery, no legacy browser support

Browser Support
Target: Last 2 versions of modern browsers (Chrome, Firefox, Safari, Edge)
No support for: IE11, legacy Edge, old mobile browsers

Utility Macros
File: templates/craft/components/utilities/classes.twig
Requirements:

Generate class strings from CMS fields
Support both old and new field structures
Validate classes
Provide fallbacks

Pattern (create this in Twig syntax):
twig{#
  Usage: {{ classes.generate(block) }}
  
  Generates utility class string from block fields
#}

{% macro generate(block) %}
  {% set classList = [] %}
  
  {# Layout utilities #}
  {% if block.gridSpan is defined %}
    {% set classList = classList|merge(['grid-span-' ~ block.gridSpan]) %}
  {% endif %}
  
  {# Typography utilities #}
  {% if block.fontSize is defined %}
    {% set classList = classList|merge(['text-' ~ block.fontSize]) %}
  {% endif %}
  
  {% if block.fontWeight is defined %}
    {% set classList = classList|merge(['font-' ~ block.fontWeight]) %}
  {% endif %}
  
  {# Color scheme #}
  {% if block.colorScheme is defined and block.colorScheme != 'default' %}
    {% set classList = classList|merge(['scheme-' ~ block.colorScheme]) %}
  {% endif %}
  
  {# Spacing #}
  {% if block.spacing is defined %}
    {% set classList = classList|merge([block.spacing]) %}
  {% endif %}
  
  {# Custom classes (power users) #}
  {% if block.customClasses is defined and block.customClasses %}
    {% set classList = classList|merge(block.customClasses|split(' ')) %}
  {% endif %}
  
  {{ classList|join(' ')|trim }}
{% endmacro %}

Critical Implementation Notes
1. Baseline System
Everything must derive from --line (the baseline height):

All spacing: calc(var(--line) * n)
Grid gaps: var(--line)
Component padding: multiples of var(--line)
Margin between elements: multiples of var(--line)

Why: Maintains vertical rhythm across the entire design.
2. Fluid Typography
Use clamp() for all type sizes:
css--text-3: clamp(1.25rem, 1.15rem + 0.5vw, 1.75rem);
Why: No breakpoint management, smooth scaling.
3. Container Queries
All components should use container queries, not media queries:
css.component {
  container-type: inline-size;
}

@container (min-width: 40rem) {
  .component { /* adjustments */ }
}
Why: Components are truly portable.
4. CSS Custom Properties for Theming
Components should use local custom properties:
css.component {
  --bg: var(--color-bg);
  --fg: var(--color-fg);
  background: var(--bg);
  color: var(--fg);
}

/* Utilities can override */
.scheme-brand .component {
  --bg: var(--color-brand-bg);
  --fg: var(--color-brand-fg);
}
5. Logical Properties
Always use logical properties:

margin-block-start not margin-top
padding-inline not padding-left/right
inset-inline-start not left

Why: International-ready by default.
6. Progressive Enhancement
The CSS should work without JavaScript. Any JS should enhance, not enable.

Design Decisions to Make
Before starting, Claude Code should ask about or make sensible defaults for:

Color palette: How many brand colors? Should it include a full palette or minimal set?
Type scale ratio: 1.2 (minor third), 1.25 (major third), 1.333 (perfect fourth), or 1.5 (perfect fifth)?
Grid columns: 12-column or 4-column system (or both)?
Measure (line length): What's the optimal characters per line? (typically 65-75ch)
Breakpoints: What container query breakpoints? (suggest: 30rem, 40rem, 60rem)

Recommended approach: Start with sensible defaults, document where they are, make them easy to customize.
